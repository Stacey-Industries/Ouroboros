# Streaming Agent-Chat State Architecture — Industry / Emerging / Cutting-Edge Spectrum

Heavy research pass to feed the Stage 1 Discovery brainstorming for Ouroboros's chat orchestration overhaul. Per `~/.claude/rules/best-practice-spectrum.md`, every topic carries **three tiers** — what most production systems do (industry standard), what's gaining adoption (emerging best practice), and what's technically advanced but high-maintenance (experimental / cutting-edge) — followed by an applicability verdict for Ouroboros's specific architecture: an Electron app driving Claude Code CLI subprocesses under subscription auth (Claude Max), with stream-json stdout and named-pipe hook events as the only "streaming server" surface.

The applicability verdicts assume the load-bearing constraints documented at the bottom of this file. Where a pattern presupposes a controllable streaming server, an SDK with structured callbacks, or an API key flow, that's flagged.

---

## Topic 1: State authority models for chat

**Industry standard — Single-source-of-truth on the server (Cursor-style, API-direct):** The server owns the message log; clients are dumb projections that read from a server endpoint and apply optimistic deltas while a turn is in flight. Cursor implements this on top of local SQLite (`state.vscdb`) that acts as the per-machine "server" — the renderer reads from SQLite, in-flight turns are streamed via the IDE's own broker, and the broker is the only writer of authoritative records. Source: [vibe-replay: What Does Cursor Store on Your Machine](https://vibe-replay.com/blog/cursor-local-storage/); [dasarpai.com: Cursor Chat Architecture](https://dasarpai.com/dsblog/cursor-chat-architecture-data-flow-storage/). **Applies to Ouroboros? Partial** — the analogous "server" for Ouroboros is the Electron main process holding the CLI subprocess; the renderer can be a pure projection, but main is the only safe authority because IPC is its native idiom and the CLI subprocess writes to it directly.

**Emerging best practice — Single-owner in one process with strict projection elsewhere (Elm/Redux unidirectional):** A single `Store` holds all chat state; every mutation is an explicit `Action`; views are pure functions of state. Used widely for chat-with-agent products because chat has many concurrent state concerns (messages, tool calls, presence, unread counters, in-flight indicators) and unidirectional flow makes ordering bugs disappear. Source: [GeeksforGeeks: Redux and Flux Architecture](https://www.geeksforgeeks.org/reactjs/redux-and-the-flux-architecture/); [Staltz: Unidirectional User Interface Architectures](https://staltz.com/unidirectional-user-interface-architectures.html). **Applies? Yes** — this is the model Ouroboros's follow-up doc proposes ("single per-thread state machine owned by main, with the renderer as a pure projection"). Reduxlike single-owner + IPC-projected renderer maps cleanly to Electron's process model and resolves the desync category that Wave 84 exhibited.

**Experimental — CRDT-backed replicated state for collaborative chat:** Yjs or Automerge as the canonical store; every window or process is a peer holding an eventually-consistent replica. Eliminates "main vs renderer disagree" by construction — there is no authority, only convergence. Yjs is the more battle-tested option; Cerbos's Playground backend uses Yjs binary blobs persisted to Postgres as the proven production shape. Source: [Velt: Best CRDT Libraries 2025](https://velt.dev/blog/best-crdt-libraries-real-time-data-sync); [Cerbos: CRDTs and Collaborative Playgrounds](https://www.cerbos.dev/blog/crdts-and-collaborative-playground); [Yjs docs](https://docs.yjs.dev/). **Applies? No (for primary chat state)** — Ouroboros is single-user, single-machine; there is no concurrent-editor problem. CRDTs cost a lot of mental overhead and trade "last-writer-wins" semantics that are inappropriate for `tool_use`/`tool_result` adjacency requirements. Reserve only if multi-window sharing of a single conversation becomes a real product requirement (and even then, an authoritative-main + broadcast pattern is simpler).

**Recommendation tier for our constraints:** Emerging — single-owner in main, renderer as pure projection. This is the pattern with the strongest fit-for-purpose match to the failure mode Wave 84 surfaced (overlapping, desynced state across processes).

## Topic 2: Event sourcing for chat specifically

**Industry standard — Event-sourced append-only log + derived view (the production trio: store + projections + snapshots):** Every state change is an immutable event appended to a per-conversation log; current state is derived by replaying the log into projections (hot conversation cache, unread counters, search index); snapshots cap replay cost. CQRS pairing is standard in 2025 — a survey of microservices teams found event sourcing + CQRS adopted by 57% of organizations. Source: [Microsoft Learn: Event Sourcing Pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing); [Baytech: Event Sourcing Explained 2025](https://www.baytechconsulting.com/blog/event-sourcing-explained-2025); [AI Academy: Event Sourcing with Kafka](https://ai-academy.training/2025/05/24/event-sourcing-with-kafka-architecture-patterns-you-should-know/). **Applies? Yes, in scaled-down form** — chat is one of the most natural fits for event sourcing because messages literally *are* events. But the heavyweight version (Kafka topics, projection workers, separate read models) is gross overkill for a single-user IDE. The valuable kernel: treat the CLI's stream-json + hook events as the canonical event log, derive UI state from a reducer over that log, and never write directly to the projection.

**Tool_use / tool_result correlation patterns — the load-bearing detail:** Anthropic requires `tool_use` blocks to be immediately followed by `tool_result` blocks in the next message; violations cause the next turn to be rejected with `tool_use ids were found without tool_result blocks immediately after`. Cross-provider agent SDKs (OpenAI Agents SDK + LiteLLM + Anthropic) have repeatedly shipped bugs where intermediate `MessageOutputItem` events get persisted between a `tool_use` and its `tool_result`, breaking the adjacency rule. Source: [openai-agents-python issue 1863](https://github.com/openai/openai-agents-python/issues/1863); [vercel/ai issue 11849](https://github.com/vercel/ai/issues/11849). **Implications for Ouroboros:** Key correlation by `tool_use.id` (`toolu_…`), maintain a `Map<tool_use.id, {accumulatedInputJson, status}>` populated as `content_block_start` events arrive, and *never persist intermediate assistant text between a tool_use and its tool_result* — this is exactly the kind of mid-stream interleaving that mid-turn render code can introduce if the projection isn't disciplined about ordering.

**Emerging — Append-only log shape but no replay (the "log as audit trail, snapshot as runtime state" hybrid):** Write events to a JSONL log for resumability and replay, but the in-memory runtime state is a normal mutable structure derived once on load. Aider, Claude Code, and Codex all use this pattern. Claude Code: `~/.claude/projects/<project>/<session-id>.jsonl` is append-only; every user message, assistant response, tool call gets a new line; compaction appends a summary entry without rewriting the file; resume replays the JSONL into in-memory state. Source: [Claude Code docs via Mintlify: history.jsonl](https://www.mintlify.com/1shanpanta/claude-analytics/reference/data-sources/history); [DeepWiki: openai/codex session management](https://deepwiki.com/openai/codex/3.3-session-management-and-persistence); [Milvus: How Claude Code Manages Local Storage](https://milvus.io/blog/why-claude-code-feels-so-stable-a-developers-deep-dive-into-its-local-storage-design.md). **Applies? Yes — this is the strongest fit.** Ouroboros already lives downstream of Claude Code's own JSONL log; mirroring the same shape internally for the IDE's projection layer means our persistence matches the CLI's, resumability is free, and there's no impedance mismatch.

**Experimental — Full CQRS with materialized read models:** Separate write path (event append) from read path (queryable projections rebuilt on demand). Source: [Mia-Platform: Event Sourcing and CQRS](https://mia-platform.eu/blog/understanding-event-sourcing-and-cqrs-pattern/). **Applies? No** — single user, single process. The "many readers diverging from one writer" problem CQRS solves doesn't exist for us. The "renderer reads from main" pattern is already the simpler version of this.

**Recommendation tier for our constraints:** Emerging — the JSONL-append-only + in-memory derived state shape. Don't replay on every read; rebuild once on load. Match Claude Code's own format where possible to keep the impedance mismatch zero.

## Topic 3: Streaming render strategies

**Industry standard — Token-level streaming with rAF batching:** Buffer incoming chunks outside React state (in a `useRef` string buffer); flush the buffer into state once per `requestAnimationFrame`. The network layer never directly drives React renders. This is the canonical pattern; render counts stay flat regardless of token rate because the buffer absorbs all variation between frames. Background tabs throttle rAF heavily (most browsers cap it around 1fps), which is desirable. Source: [SitePoint: Streaming Backends & React](https://www.sitepoint.com/streaming-backends-react-controlling-re-render-chaos/); [DEV: Chasing 240 FPS in LLM Chat UIs](https://dev.to/gokhan_koc_88338a026508b3/chasing-240-fps-on-llm-chats-4gde); [Pockit/DEV: Streaming LLM Responses Guide](https://dev.to/pockit_tools/the-complete-guide-to-streaming-llm-responses-in-web-applications-from-sse-to-real-time-ui-3534). **Applies? Yes** — this is straightforwardly applicable to the renderer regardless of the streaming source.

**Industry standard — Virtualized list rendering for long conversations:** TanStack Virtual / react-virtuoso with dynamic `measureElement` for variable-height bubbles, reserve index 0 for a loading sentinel, IntersectionObserver on a top sentinel to trigger prepend, scroll-offset capture/restore on prepend, `React.memo` on the message bubble component, `overscan` of 5–10. Source: [LogRocket: TanStack Virtual](https://blog.logrocket.com/speed-up-long-lists-tanstack-virtual/); [TanStack Virtual Infinite Scroll Docs](https://tanstack.com/virtual/v3/docs/framework/react/examples/infinite-scroll); [Medium: Reverse Infinite Scroll with TanStack Virtual](https://medium.com/@rmoghariya7/reverse-infinite-scroll-in-react-using-tanstack-virtual-11a1fea24042). **Applies? Partial** — Ouroboros's chat surface has tool-call panels, diff blocks, and rich content that vary wildly in height; dynamic measurement is mandatory. Virtuoso has first-class reverse-scroll support and is often the path of least resistance for chat UIs. The ~100-session accumulation problem flagged in the follow-up doc is partly a virtualization gap, partly a pruning gap.

**Emerging — `useSyncExternalStore` over the buffer, with `startTransition` for non-urgent appends:** React 18's external-store hook lets the buffer live entirely outside React's state model; `startTransition` marks streaming updates as interruptible so input remains responsive even during heavy bursts. Combine with rAF batching, not as a replacement. Source: [SitePoint, same as above]; [DEV: Chasing 240 FPS].  **Applies? Yes** — `useSyncExternalStore` is the "React-correct" way to subscribe a component to an external mutable buffer; it composes naturally with a main-owned reducer projected through preload.

**Emerging — Backpressure handling at the IPC boundary:** When the renderer can't keep up with the subprocess emission rate, the standard approach is per-channel buffering limits + explicit pacing. For an Electron app, the analog is "if rAF flush is lagging, accumulate more in the buffer before flushing — don't queue IPC messages faster than the renderer can drain them." Source: [Vertext Labs: WebSocket backpressure for chat streams](https://vertextlabs.com/websocket-backpressure-flow-control-real-time-chat-streams/); [Microsoft Tech Community: Streaming for LLM chat apps](https://techcommunity.microsoft.com/blog/azuredevcommunityblog/the-importance-of-streaming-for-llm-powered-chat-applications/4459574). **Applies? Yes — critical** — the Wave 84 Phase D investigation found that bridge or subprocess buffering was producing only 5 emit events across a 145s turn for content that should stream incrementally. This is the backpressure problem manifesting as the *opposite* failure: not "renderer can't keep up," but "main isn't flushing because it's batching too aggressively." Either direction of backpressure mishandling produces the same UX symptom.

**Experimental — Optimistic UI for in-flight tool calls driven by streaming tool-call arguments:** Microsoft's AG-UI integration with the .NET Agent Framework streams tool arguments as `STATE_DELTA` events while the LLM is still generating them, letting the UI render a predictive view ("the agent is about to call `edit_file` with these args") before the tool actually runs. Final `STATE_SNAPSHOT` reconciles. Source: [Microsoft Learn: AG-UI State Management](https://learn.microsoft.com/en-us/agent-framework/integrations/ag-ui/state-management); [AG-UI Protocol Docs](https://docs.ag-ui.com/introduction). **Applies? No (today), but worth noting** — Anthropic's stream-json does emit `input_json_delta` partial JSON during a `tool_use` block, so the *data* exists. But Ouroboros has no current UI affordance for "the agent is about to call X" optimistic preview, and adding one is a feature, not a state-architecture fix. Park it.

**Recommendation tier for our constraints:** Industry standard for rAF + virtualization; Emerging for `useSyncExternalStore` + backpressure-aware flushing.

## Topic 4: Identity / correlation patterns

**Industry standard — Hierarchical IDs (conversation_id → message_id → tool_use_id):** Three layers of stable identifiers, each scoped to its level. The `thread_id` / `conversation_id` outlives the run; the `run_id` / `message_id` scopes a single turn; the `tool_use_id` scopes one tool invocation within a turn. AG-UI codifies this — thread IDs persist across runs; run IDs are per-execution; tool_call IDs correlate tool args with results. Source: [AG-UI Protocol Docs](https://docs.ag-ui.com/introduction); [CopilotKit: AG-UI Protocol](https://www.copilotkit.ai/blog/ag-ui-protocol-bridging-agents-to-any-front-end). **Applies? Yes — directly addresses Ouroboros's current pain.** The Wave 84 framing flagged at least three session-ID namespaces (hook-pipe UUID, stream-json UUID, internal thread UUID). The right architectural response is to *not* try to make them the same — instead, formalize them as different *levels* of identity with explicit aliasing.

**Emerging — Client-generated keys for optimistic-record matching:** Server-assigned `_id` won't match the client's optimistic record, so reconciliation has to happen on a separately stable key. Convex Agent's pattern: optimistic records and persisted records share `${threadId}-${order}-${stepOrder}` even though `_id` differs. Source: [DeepWiki: Convex Agent Optimistic Updates](https://deepwiki.com/get-convex/agent/7.5-optimistic-updates). **Applies? Partial** — Ouroboros doesn't have a "server" generating IDs; the CLI does. The CLI's `--resume` UUID is the closest thing to a stable thread ID, but it's CLI-issued, not user-issued. The pattern that translates: when the IDE assigns its own thread/message IDs internally (because the CLI hasn't emitted its first event yet), record an explicit alias to the CLI's IDs when they arrive, instead of trying to retroactively rewrite the IDs everywhere.

**Emerging — Idempotency keys for retries:** Stripe-style idempotency keys on the send path; if the same key arrives twice, the second is a no-op. Critical when retries can occur (network failure mid-send, app reload mid-send). Source: [Stardrift: Resumable LLM streaming](https://stardrift.ai/blog/streaming-resumptions); [Ably: Resume tokens and last-event IDs](https://dev.to/ablyblog/resume-tokens-and-last-event-ids-for-llm-streaming-how-they-work-what-they-cost-to-build-4l7e). **Applies? Partial** — retries on the *send* path are rare for Ouroboros (the CLI is local; "the network" is a pipe). But retries on the *render* path (re-mounting a chat after a window operation) absolutely happen, and an idempotency key on event consumption ("have I already applied this stream-json event?") prevents double-application bugs.

**Experimental — Content-addressed IDs (hash-of-content as identity):** Treat the canonical ID of a message as the hash of its content, so identical messages from any source collapse to the same record. Used heavily in event-sourced systems and git-like architectures. **Applies? No** — chat messages are timestamped, distinct even when content matches, and content-addressing adds complexity without solving the actual problem (which is conflation between *namespaces*, not duplication within one).

**Recommendation tier for our constraints:** Industry standard — formalize hierarchical IDs (CLI thread UUID → CLI message ID → tool_use_id) and add an explicit *alias table* in main that maps Ouroboros's internal IDs (assigned at chat-tab creation, before CLI emits anything) to CLI IDs (assigned when the CLI starts streaming). Never try to make them be the same value.

## Topic 5: Session lifecycle / teardown

**Industry standard — Explicit state machine (`created → active → completed → pruned`):** Every session has a documented terminal transition. The dev.to investigation of "when your agent slowly eats all the memory" identified the absence of the final transition as the root cause of accumulation leaks: sessions are created and stay forever because nothing prunes them. Source: [DEV: When Your Agent Slowly Eats All the Memory](https://dev.to/oolongtea2026/when-your-agent-slowly-eats-all-the-memory-5al1). **Applies? Yes — directly addresses the ~100-session-accumulation problem flagged in the Wave 84 follow-up.** The fix isn't "add a cleanup timer," it's "make the terminal state explicit and implement the transition." When is a session really over? The standard answer: when the CLI subprocess exits AND the user is no longer viewing that thread AND a configured retention period has elapsed. Three signals, not one.

**Industry standard — Owner-aware teardown:** Whoever creates a resource owns its `close()`. OpenAI Agents SDK's pattern: `from_uri(...)` creates and owns the MongoDB client and closes it; `MongoDBSession(client=...)` makes `session.close()` a no-op because lifecycle stays with the caller. Source: [OpenAI Agents SDK Sessions](https://openai.github.io/openai-agents-python/sessions/). **Applies? Yes** — for Ouroboros, the chat subsystem owns the CLI subprocess (it spawned it); session teardown means it kills the subprocess. The renderer never owns the subprocess; renderer-side cleanup is pure projection cleanup (drop the buffer, clear the in-flight indicator).

**Emerging — Resumable streams with stream/run/connection separation:** Decouple generation from delivery. The LLM generation runs in its own process and writes to durable storage (Redis Streams, Workflow runID); the client connection is just a consumer that can come and go. App reload mid-stream → reconnect to the same `runID` and resume from the last delivered offset. Source: [Upstash: Resumable LLM Streams](https://upstash.com/blog/resumable-llm-streams); [Ably: Resumable LLM streaming](https://ably.com/blog/resumable-llm-streaming-session-recovery-across-reconnects); [LibreChat: Resumable Streams](https://www.librechat.ai/docs/features/resumable_streams). **Applies? Yes** — this is exactly Ouroboros's architecture by accident: the CLI subprocess is the "generation process running independently"; main is the "broker that consumes stdout"; renderer is the "client that can reload." The pattern says: on app reload, reconnect to the in-flight CLI subprocess (if it's still alive) and replay the event log from offset N. Don't kill the CLI just because the renderer reloaded.

**Emerging — Orphaned-tool-call auto-healing:** A pernicious failure mode: process crashes mid-tool-call leave the conversation history with a `tool_use` that has no matching `tool_result`. The next LLM call rejects the request and the session becomes permanently un-resumable. Both Google ADK ([issue 3971](https://github.com/google/adk-python/issues/3971)) and pi-mono ([issue 2119](https://github.com/badlogic/pi-mono/issues/2119)) document this. The mitigation: when loading a session, detect dangling `tool_use` blocks and inject a placeholder `tool_result` ("Error: Execution interrupted") so the LLM understands the prior action failed and can continue. **Applies? Yes — this is a real risk for Ouroboros** any time the CLI subprocess crashes or is killed mid-tool. The auto-heal pattern should be implemented in the session-load path.

**Experimental — Durable workflow runtimes (Restate, Temporal-style):** Treat each agent run as a durable workflow that survives process crashes via promise persistence. Source: [Restate: Building Durable AI Agents](https://www.restate.dev/blog/building-durable-agents-with-vercel-and-restate). **Applies? No** — Ouroboros runs the CLI; the CLI isn't a durable workflow, and we don't control it. The cost of running a durable workflow runtime inside an Electron app for the IDE's own state isn't justified.

**Recommendation tier for our constraints:** Industry-standard explicit state machine + owner-aware teardown + emerging-tier orphaned-tool-call auto-healing. The resumable-streams pattern is appealing but requires the CLI to support `--resume` correctly (it does), so it's achievable; treat it as a design goal, not a Phase 1 requirement.

## Topic 6: Multi-process state synchronization in Electron

**Industry standard — Main owns state, renderer is pure projection, broadcast on mutation:** Main holds the canonical store; renderer mutations route through `ipcRenderer.invoke('chat:send', …)` → `ipcMain.handle`; main applies the mutation and broadcasts via `webContents.send` on a per-key or per-slice channel; renderers consume via `useSyncExternalStore`. Pull-on-demand snapshots for cold load; revision-gated re-fetch during bursts to avoid IPC churn. Source: [Bruno Scheufler: Syncing State between Electron Contexts](https://brunoscheufler.com/blog/2023-10-29-syncing-state-between-electron-contexts); [BigBinary: Synchronized store between main and renderer](https://www.bigbinary.com/blog/sync-store-main-renderer-electron); [state-sync: Electron any-store guide](https://777genius.github.io/state-sync/examples/electron-any-store). **Applies? Yes — this is the target architecture** for the chat overhaul, exactly as the follow-up doc described.

**Emerging — Zubridge / electron-redux-style typed bridges with delta updates and action batching:** Libraries that wrap the main-as-owner pattern with built-in optimization: action batching (groups renderer actions within a configurable window into single IPC calls) and delta updates (sends only changed portions of the state tree instead of the full tree). Source: [@zubridge/electron on npm](https://www.npmjs.com/package/@zubridge/electron). **Applies? Yes, but probably not adopt-as-is** — the bridges are well-engineered, but they impose their own store shape (typically Zustand-flavored). For a chat-specific state machine that needs strict ordering guarantees on `tool_use`/`tool_result` pairs, custom IPC contracts with explicit channels are easier to reason about than a generic state-sync library.

**Emerging — MessagePort-based direct renderer-to-renderer for hot paths:** Electron's `MessageChannelMain` allows wiring two webContents together so they exchange messages without main-process serialization overhead. Useful for streaming high-frequency tokens to a renderer without main acting as a relay on every frame. Source: [Electron MessagePorts docs](https://www.electronjs.org/docs/latest/tutorial/message-ports). **Applies? Probably no** — Ouroboros's "streaming" already goes main → renderer (CLI stdout is in main). MessagePorts would help if a renderer were sourcing the stream and broadcasting to other renderers, which isn't our shape. Reserve for if multi-window same-conversation becomes a feature.

**Experimental — Event-sourced replication across IPC:** Treat IPC as a transport for an event log; every process maintains its own derived state by replaying the log. Source: implicit in event-sourcing literature ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)). **Applies? No** — the cost of every renderer maintaining its own reducer-driven projection exceeds the benefit when main can just send the projected state directly.

**Recommendation tier for our constraints:** Industry standard — main owns, renderer projects, broadcast on mutation, pull-on-demand for cold load. Use a custom typed bridge (not a generic library) because the chat state machine has domain-specific ordering guarantees that justify hand-rolled IPC contracts.

## Topic 7: Persistence layer choices for chat history

**Industry standard — SQLite per-machine (Cursor, VS Code generally):** SQLite via VS Code's `ItemTable` + `cursorDiskKV` key-value pattern; `bubbleId:<composerId>:<bubbleId>` keys for individual messages, `composerData:<composerId>` for session metadata. Split global DB (message content) vs workspace DB (sidebar pointers). Source: [vibe-replay: Cursor Local Storage Deep Dive](https://vibe-replay.com/blog/cursor-local-storage/); [Cursor.fan: Chat history management](https://www.cursor.fan/tutorial/HowTo/manage-cursor-chat-history/). **Applies? Yes** — and Ouroboros already uses SQLite (`sessionsData` for window sessions). Extending SQLite to be the durable chat-history store is a low-friction continuation.

**Emerging — JSONL append-only files (Aider, Claude Code, Codex):** One file per session under a per-project directory; each line is a complete event; append-only with crash safety (at most one partial trailing line is lost); compaction appends a summary entry rather than rewriting. The format is uniquely well-suited to interactive AI agents: many small sequential writes, occasional full-file reads on resume, no transactional overhead. Source: [DataScienceInAction: Finding Claude Code Chat History](https://datascienceinaction.substack.com/p/finding-your-claude-code-chat-history); [DeepWiki: openai/codex session management](https://deepwiki.com/openai/codex/3.3-session-management-and-persistence); [Milvus: How Claude Code Manages Local Storage](https://milvus.io/blog/why-claude-code-feels-so-stable-a-developers-deep-dive-into-its-local-storage-design.md); [Claude Code SDK: Persist sessions](https://code.claude.com/docs/en/agent-sdk/session-storage). **Applies? Yes — strongly** — Claude Code already writes JSONL files at `~/.claude/projects/<project>/<session-id>.jsonl`. Ouroboros could either (a) read those files directly as the source of truth for resume, (b) mirror to its own JSONL format for IDE-specific metadata, or (c) use SQLite for index + JSONL for body. Option (c) is the most common production shape — SQLite indexes the conversation list and metadata; JSONL holds the actual message stream for replay.

**Emerging — Hybrid SQLite (index/metadata) + JSONL (message bodies):** SQLite indexes `(thread_id, project, last_active, title)`; JSONL holds the full event stream. Best of both: fast list queries, append-only safety on the hot path, no row-rewrite contention on long conversations. **Applies? Yes** — this is the recommended target.

**Experimental — IndexedDB-backed CRDTs for offline-first sync:** Used in web-shipped IDEs that need to sync the same conversation across devices. Source: [DEV: Building Offline-First Collaborative Editors with CRDTs and IndexedDB](https://dev.to/hexshift/building-offline-first-collaborative-editors-with-crdts-and-indexeddb-no-backend-needed-4p7l). **Applies? No** — Ouroboros is Electron, not web; cross-device sync isn't a current product requirement; IndexedDB's strengths are wasted on a single-user single-machine app.

**Retention / pruning policy norms:** TTL on the persisted conversation store itself is commonly omitted, leading to accumulation. Spring AI's pattern: a `memoryConsolidationTrigger` predicate that auto-runs a consolidation pass when fired. The `agentmemory` project uses Ebbinghaus-curve decay. Source: [Spring AI: Agentic Patterns 6 — AutoMemoryTools](https://spring.io/blog/2026/04/07/spring-ai-agentic-patterns-6-memory-tools/); [agentmemory on GitHub](https://github.com/rohitg00/agentmemory). For Ouroboros: a configurable retention window per project, an "archive instead of delete" option for compliance-minded users, and an explicit pruner that runs on app startup (not on a timer mid-session).

**Recommendation tier for our constraints:** Emerging — SQLite index + JSONL bodies, with a documented retention policy. Read Claude Code's JSONL directly for the CLI's view; mirror to our own JSONL when we have IDE-specific metadata to attach.

## Topic 8: Multi-window / multi-tab state ownership

**Industry standard — Per-window isolation with shared authoritative state in main:** Each `BrowserWindow` has `contextIsolation: true` and its own preload; main holds the authoritative chat state; mutations from any window go to main; main broadcasts via `BrowserWindow.getAllWindows().forEach(w => w.webContents.send(...))` excluding the sender. New windows hydrate by calling `ipcRenderer.invoke('chat:get-history')`. Per-window UI state (scroll position, draft text, filter selections) lives only in that renderer. Source: [Electron IPC Tutorial](https://www.electronjs.org/docs/latest/tutorial/ipc); [Bloomca blog: Multiple Windows in Electron Apps](https://blog.bloomca.me/2025/07/21/multi-window-in-electron.html). **Applies? Yes** — matches Ouroboros's existing per-window project-roots model; extending the same shape to per-window chat-tab subscriptions is the natural fit.

**Emerging — BroadcastChannel for renderer-to-renderer same-origin sync:** When all chat windows share an origin, `BroadcastChannel` lets renderers gossip directly without going through main. Best for derivative UI state (e.g., "this thread is now active in Window A so dim its tab in Window B") that doesn't need authoritative ordering. Source: [Bloomca, same as above]. **Applies? Partial** — derivative cross-window UI signals are a legitimate use, but anything authoritative still routes through main. Keep BroadcastChannel as an optional optimization, not as the primary sync mechanism.

**Emerging — "Same conversation, two windows" via subscription to a shared store:** Two windows showing the same thread both subscribe to the same main-owned reducer; the chat-tab UI state is per-window, but the underlying conversation events are shared. Edits in one window appear in the other near-instantly. **Applies? Yes if/when this becomes a product requirement** — the architecture being recommended (main-owned reducer, renderer projection) naturally supports this without special handling. Just don't *prevent* it; the projection model makes it free.

**Tab close vs window close vs app close — distinct lifecycle events:**
- Tab close: drop renderer subscription, keep main's session record (user might reopen the tab).
- Window close: drop all of that window's renderer subscriptions; if no other window holds the session, transition the session toward archival.
- App close: persist all in-flight sessions to disk; close subprocesses cleanly with their `tool_use`/`tool_result` pairs intact (auto-heal on next load if not).

The Wave 84 follow-up flagged that "lifecycle events are fan-out without a single owner — events flow to multiple reducers, multiple contexts, multiple panels, with no canonical 'the chat session ended, drop everything related to it' path." This is the failure mode the explicit-state-machine pattern (Topic 5) directly solves.

**Experimental — Window-as-actor in an XState-style hierarchy:** Each window is an actor child of a root chat-session actor; window lifecycle is just actor lifecycle; `actor.stop()` cascades cleanup. Source: [Stately: Actors](https://stately.ai/docs/actors); [This Dot Labs: XState Async Workflows](https://www.thisdot.co/blog/using-xstate-actors-to-model-async-workflows-safely). **Applies? Worth considering** — the parent-child invocation discipline (`fromCallback` for the streaming actor, auto-disposed on parent state exit) directly addresses the "no canonical teardown path" problem. But the cost is a full XState adoption, which is a much bigger commitment than the rest of the recommendations. Park it as an option to revisit if the hand-rolled state machine starts duplicating XState semantics anyway.

**Recommendation tier for our constraints:** Industry standard — per-window isolation, main-owned authoritative state, hydrate-on-window-open. Document the three lifecycle events (tab close / window close / app close) as distinct transitions. Park the XState option but keep it on the radar.

## Synthesis: load-bearing constraints

Most chat-architecture blog posts assume you control the streaming server. Ouroboros doesn't. These are the constraints that change which patterns apply:

1. **The streaming source is a subprocess we don't own.** The Claude Code CLI is the producer; we are downstream consumers of its stdout and named-pipe hook events. We cannot add fields to the wire format, change the event schema, or guarantee event ordering between channels (stdout vs hook pipe vs in-process synthetic events have *no* cross-channel ordering guarantee). Any architecture that assumes "we control what events fire and in what order" is wrong for us. Implication: the event-log shape must be defined defensively, accept events from multiple sources, and apply ordering rules in the reducer rather than relying on producer discipline.

2. **Three identity namespaces, none of which we issue.** The CLI issues the `--resume` UUID; the hook pipe issues its own session identifier on the named-pipe handshake; the stream-json event headers carry their own `session_id`. Plus our IDE-internal thread ID, assigned at chat-tab creation before the CLI has even started. Architectures that assume "one ID per session" are wrong for us. The right pattern is an explicit alias table (Topic 4), not retroactive ID rewrites.

3. **No SDK callbacks; only stream events and pipe events.** Direct API and Agent SDK patterns assume structured callbacks (`onToolCall`, `onMessageComplete`, etc.). We have raw stream-json deltas and named-pipe hook events, both of which require parsing and correlation before they're useful. Implication: the "normalize at an abstraction boundary" pattern (Topic 2) is mandatory — there must be a layer in main that converts CLI events into our canonical event log, and the rest of the system reads only from that canonical log.

4. **No prompt caching, no countTokens, no message-history retrieval API.** The CLI manages its own context; we cannot inspect or control what it sends to Anthropic. Implication: Ouroboros's "conversation history" is *its own* derived projection of what it observed, not a queryable representation of what the model actually saw. We must not assume the IDE's projection is the same as the CLI's internal state. When in doubt, the CLI's JSONL log under `~/.claude/projects/` is the closer-to-canonical record.

5. **Hook events and stream-json events are two independent streams about overlapping facts.** A `session_end` event might arrive via the hook pipe; the stream-json might still emit final deltas after it; the IDE's main process sees both. Wave 84 Phase A's investigation found this exact race. Implication: the reducer must be idempotent and order-tolerant; suppression on one channel cannot assume the other channel has also stopped emitting. Patterns that assume "one event marks one transition" are wrong; we need "two-channel reconciliation with explicit precedence."

6. **The terminal Claude Code session — the one editing Ouroboros itself — is always running.** The IDE runs inside itself. Hook events from the terminal session interleave with hook events from the chat agent's session on the same pipe. Implication: every reducer that consumes hook events must filter by session ID, and the alias table from constraint #2 must include "this hook event belongs to which IDE thread, if any." Patterns from documents that assume "the only chat session is the chat session" are wrong by default.
