# Wave 82 — Chat-Only Polish Bundle

## Status

DRAFT · target v2.13.0 · drafted 2026-05-03.

## Context — why this wave exists

A focused user-driven bug-hunt session on 2026-05-03 surfaced 15 distinct defects across the chat-only / workbench surfaces of the Agent IDE. The bugs span four functional areas: (1) chat-only / workbench title-bar wiring (most menu items dispatch DOM events that nothing listens for — `WORKBENCH_NEW_SESSION_EVENT`, `WORKBENCH_NEW_CHAT_EVENT`, `WORKBENCH_OPEN_PROJECT_EVENT`, `WORKBENCH_SWITCH_PROJECT_EVENT` and likely most of View / Tools / Help / Edit Find-Next/Prev have zero `addEventListener` consumers, verified by grep across `src/renderer`); (2) state-management cascades (chat-delete in chat-only view causes a ~0.2s rail flash because the sidebar mutates the Zustand store directly while the workspace's `useThreadState` is unaware — `ChatHistorySidebar.tsx:258-264` vs `agentChatWorkspaceActions.ts:116-135`; outer-rail projects don't persist because `OuterProjectRail.useAddProject` writes only per-window state but not `config.recentProjects`, and chat windows initialize with `projectRoots: []` per `windowManager.ts:244` with no restore path); (3) artifact-pane / file-viewer issues (Recent section auto-populates on every observation, `ArtifactPaneHeader` is redundant with the tab row, edit-mode causes a re-mount that nukes toolbar state and disappears the Edit button, minimap renders alongside the default scrollbar, blame/outline toggles do nothing visible); and (4) context-preview popover gaps (project rules don't load even when `.claude/rules/` is populated, Skills tab shows only executions never available skills, Files tab semantics are unclear, file-tree heat map button is purely cosmetic because `useFileHeatMap.extractFilePath` rejects every JSON tool input).

Many of these are tied to the chat-only / workbench shell that was built incrementally across waves 42-47, 59. Several represent honest follow-ups from those waves (per `roadmap/follow-ups/follow-ups.md`); some are net-new defects from rapid iteration; one (composer typing lag during rule-load bursts) is a perf cascade that touches the shared `AgentEventsContext` so its blast radius extends beyond chat-only. Every bug has been root-caused or scoped to a specific code site through static analysis; only four require runtime profiler / instrumented diagnosis before the fix shape can be locked.

This wave is structurally unusual — not a single-feature build but a coordinated cleanup pass. Phases are organized by **bug class** (mechanical / state / wiring / diagnostic) rather than by build-order, with explicit parallelism across independent surfaces. Phase 0 captures four product-decision items that can't be locked from grounding alone (Recent-section eviction policy, Timeline tab disposition, Files tab semantics, Skills tab data source); the remaining decisions are locked from the user's stated preferences during the bug-hunt session.

## Goal

After this wave, the chat-only / workbench shell behaves as the user expects across the surfaces touched in the 2026-05-03 bug hunt: title-bar menu items either fire their intended action or are removed; deleting a chat causes no rail flash; projects added via the outer rail persist across window restarts; the artifact pane edit-mode toolbar survives entering and exiting edit mode; the file-tree heat map actually colors rows when the agent edits files; the context-preview popover loads project rules reliably and surfaces skills meaningfully; the composer no longer lags during agent rule-load bursts. No new features ship; the wave is purely closing the gap between what the surfaces look like and what they actually do.

## Locked decisions (Phase 0 — ADR)

ADR file: `roadmap/wave-82-DRAFT/wave-82-decisions.md`.

**Locked from grounding (user statements during 2026-05-03 bug hunt):**

1. **Heat map fix path: extend `extractFilePath` to JSON-parse structured tool inputs and read `parsed.file_path` (Write/Edit) or `parsed.notebook_path` (NotebookEdit), falling back to the raw-string path for legacy-shape inputs.** Industry-standard for tool-input parsing in 2026 LLM tooling — every Anthropic / OpenAI tool-use spec since mid-2024 returns structured JSON. The original string-only path is dead for real tool calls.
2. **Branch indicator removal: drop `<GitSection>` mount from `StatusBar.tsx:208`** (or specifically `BranchButton` at line 113); keep `GitBranchIndicator` at top of file tree (`FileTree.tsx:243`). User stated preference for the file-tree variant.
3. **Chat-delete cascade fix: route both `ChatHistorySidebar.handleDelete` and `useWorkbenchRailActions.onDeleteThread` through `model.deleteThread` (the canonical workspace action wired into the store via `useSyncActionsIntoStore`)**, eliminating the direct `applyDeleteToStore` call. The workspace's internal `useThreadState` is the source of truth; the sidebar must use it, not race against it.
4. **Outer-rail persistence quick fix: mirror the IDE-shell add path — when `OuterProjectRail.useAddProject` runs, also write to `config.recentProjects`** (matching `InnerAppLayout.tsx:174-178` and `useProjectManagement.ts:28-35`). Restores the round-trip via the existing fallback merge in `useWorkbenchProjects` at `ChatWorkbenchBody.rails.tsx:37`. Structural fix (per-window roots persisted to sessionsData/SQLite) deferred — see "Out of scope."
5. **Activeproject desync handling: validate `layout.activeProject` against the merged project list on load; if not present, clear it.** One-liner; prevents the inner rail showing a project that's no longer in the outer rail.
6. **Edit-mode label: rename "Exit Edit" to "Exit"** in `FileViewerToolbar.tsx:224`. One-line label change.
7. **Workbench menu wiring strategy: subscribe each unwired event at the `ChatWorkbenchShell` or `TwoTierRailSurface` level** so the listener has access to `layout`, `handlers`, `terminal`, etc. Existing handler implementations (`handleCreateSession`, `model.startNewChat`, `addProjectRoot`, `layout.setActiveProject`) are reused; the wiring is purely subscription, not new logic.
8. **Terminal close affordance: add hover-revealed `TabCloseButton` to `InnerSidebarTerminals` rows AND a "Close session" action button in `ChatWorkbenchTerminalDock` header next to "+ New".** Both bind to `terminal.handleTerminalClose`. The dock header's existing ✕ button stays — it closes the dock as documented.

**REQUIRES USER LOCK (4 items — recommended answers below per `~/.claude/rules/best-practice-spectrum.md`):**

9. **Artifact pane Recent section semantics.**
   - _Industry-standard (recommended pick):_ Recent section caps at 5 chips per row with up to 2 rows (10 total); current open-files row caps at 5; on overflow, oldest open file moves to Recent and the underlying `OpenFile` stays loaded in `FileViewerManager` but is hidden from the active tab strip; clicking a Recent chip moves it back to the open-files row, evicting the oldest open file to Recent (LRU swap). Cap applies to file artifacts only (diffs are transient, surface-once). Source: VS Code's "Recently Opened Editors" pattern (Ctrl+Shift+T) — proven LRU + cap UX.
   - _Emerging:_ Don't unload at all — let `useFileViewerManager` carry unbounded open files but only display 5 in the tab row; Recent is just an alternative view of the rest.
   - _Experimental:_ MRU pinning — user explicitly pins which 5 stay in the open row; Recent shows everything else.
   - REQUIRES USER LOCK.

10. **Timeline tab disposition.**
    - _Industry-standard (recommended pick):_ Lift the cap to ~500 entries (paginated "Load more" affordance for older), keep the tab as-is. Cheapest change; preserves a working surface; counter and visible list stop disagreeing.
    - _Emerging:_ Repurpose Timeline as a digest view (collapsed by session, file-change summaries, milestone markers — not raw events); Monitor stays the raw stream. Differentiates the two surfaces meaningfully.
    - _Experimental:_ Drop Timeline entirely — Monitor covers it.
    - REQUIRES USER LOCK.

11. **Files tab semantics in context-preview popover.**
    - _Industry-standard (recommended pick):_ Unify "Files" and "Mentions" into a single "Context" tab grouped by source (pinned / mentioned / dropped / pasted images / pasted text). Single mental model: "what's going to the prompt." Removes the current confusion (mentions ≠ files ≠ attachments).
    - _Emerging:_ Keep tabs separate but add an "Attachments" tab for images/binaries; Files = pinned only; Mentions = `@`-typed only.
    - _Experimental:_ No tabs at all; flat scrollable list with type badges.
    - REQUIRES USER LOCK.

12. **Skills tab data source.**
    - _Industry-standard (recommended pick):_ Show available skills (the long catalog in the system reminder) as the primary list, with a small "Recently executed" section at the top when `skillExecutions[]` is non-empty. Mirrors the Rules tab's User/Project sub-grouping. Requires a new `rulesAndSkills:listSkills(projectRoot)` IPC analogous to `listRuleFiles`.
    - _Emerging:_ Show only executions but make the empty state actionable ("X skills are available — try /<command>"). Cheapest.
    - _Experimental:_ Defer the tab entirely; remove from popover until skills become a more prominent feature.
    - REQUIRES USER LOCK.
