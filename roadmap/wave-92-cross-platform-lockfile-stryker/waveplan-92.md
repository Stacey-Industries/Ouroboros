# Wave 92 — Cross-Platform Lockfile Foundation + Stryker Activation

## Status

DRAFT · target `v2.17.0` · drafted 2026-05-16.

## Context — why this wave exists

Agent IDE has no current cross-platform lockfile divergence (confirmed during Wave 88's 7-round CI fix-sweep: Round 1 ruled out lockfile divergence as the cause; Rounds 2-7 fixed unrelated test-bug surfacing). **Installing Stryker is the trigger that would introduce divergence** — its `vitest-runner` dependency tree historically pulled `jsdom` → `@asamuzakjp/css-color` (optional, platform-specific), and any future native-binding devDep (`esbuild`, `rollup`, `sharp`, `lightningcss`) would do the same. This wave adopts the foundation tooling *preventatively* so the trigger lands on a hardened surface, then installs Stryker on top.

The foundation pattern (`lockfile:sync` wrapper, pre-push provenance guard, CI canary, Stryker dual-trigger workflow, anti-backslide `break:` floor) is **shipped and proven in Gamify Wave 9** (`C:\Web App\Gamify\roadmap\wave-9-cross-platform-lockfile-stryker/`, SHIPPED 2026-05-15). All concrete deliverables — `stryker.config.mjs`, `ci-stryker.yml`, `lockfile-sync.mjs`, and two vendor-gotchas files — exist on disk and are the canonical reference. This wave's job is **inherit the pattern, apply Agent IDE deltas, install Stryker for the first time, ship**. The meta-initiative framing was retracted at Gamify Wave 9 wave-end per Cole; each repo handles its own.

The Agent IDE deltas that justify a real wave (rather than mechanical copy): (a) **Electron native modules.** Agent IDE imports `better-sqlite3` (`src/main/storage/database.ts:12`), `node-pty` (`src/main/pty.ts:2`), and `@parcel/watcher` (`src/main/watchers/nativeWatcher.ts:10`); Stryker's sandbox cannot rebuild native modules ([stryker-js#1621](https://github.com/stryker-mutator/stryker-js/issues/1621)), so `mutate` globs must be scoped to pure-logic code that does not import these — even transitively. The codebase scan found each native dep has a single import site already, so the "adapter refactor" is "harden the existing seam + audit for direct imports," not "build adapter from scratch." (b) **`@node-rs/xxhash` is a dead dep** — declared in `package.json:109`, zero imports across the codebase; gets removed during Phase 4's `overrides` audit. (c) **Engine pin is Node ≥20** (`package.json:engines.node`), not Gamify's 22; the WSL2 lockgen pipeline runs against Node 20.20.2 (already installed via nvm from Gamify's pre-wave setup). (d) **CI matrix is 3-OS** (ubuntu/windows/macos) where Gamify's is ubuntu-only — the CI canary must validate all three; the existing `ci.yml` already runs the full test suite per-OS, so the canary layers cleanly. (e) **Soft dependency met**: Agent IDE's red-master CI bug (`roadmap/bugs/2026-05-14-master-ci-ubuntu-windows-failures.md`) was RESOLVED 2026-05-15 across 7 commits; master CI is green on all three platforms with e2e step disabled (separate `roadmap/bugs/2026-05-15-e2e-teardown-hang.md`, out of wave scope).

## Goal

After Wave 92, Agent IDE has a single `npm run lockfile:sync` command that regenerates `package-lock.json` via WSL2-native generation — Cole never opens a raw WSL2 terminal. A git pre-push hook blocks any `package-lock.json` change lacking the `lockfile:sync` provenance marker. A CI canary validates the committed lockfile resolves on win32 + linux + darwin on every PR that touches it. The three live native-module imports (`better-sqlite3`, `node-pty`, `@parcel/watcher`) are confirmed to have single-import-site adapter shapes with no leaks; `@node-rs/xxhash` is removed as a dead dep. Stryker 9.6.x is installed with `mutate` globs scoped to pure-logic code (`src/shared/**`, expand from there), the first mutation baseline measured, and `break:` set just below it as an anti-backslide floor. A `ci-stryker.yml` workflow runs incremental on PR + push-to-master and a full `--force` run on a weekly cron. `.stryker-tmp/` (already gitignored) and `reports/stryker-incremental.json` (new) are out of git. `.nvmrc` exists at repo root (`20`). The two Gamify vendor-gotchas (`stryker.md`, `wsl2-lockgen.md`) are promoted into Agent IDE's `.claude/vendor-gotchas/` with sdkVersion/relatedPaths adjusted, plus a new `stryker-electron.md` capturing the native-bindings `mutate` glob discipline this wave is the first to prove.

## Locked decisions (Phase 0 — ADR)

ADR file: `roadmap/wave-92-cross-platform-lockfile-stryker/wave-92-decisions.md`.

1. **Inherit Gamify Wave 9's pattern wholesale.** `lockfile-sync.mjs`, `ci-stryker.yml`, `stryker.config.mjs`, pre-push guard, vendor-gotchas — all start as direct copies of Gamify's shipped files, with Agent IDE deltas applied (Node 20, master branch, single-manifest, electron-rebuild interaction). Net-new design only where Agent IDE materially differs from Gamify (Electron native-module sandbox-isolation; CI canary across 3-OS).
2. **Lockfile generation runs as a full from-scratch `npm install --ignore-scripts --no-audit --no-fund` in WSL2-native `$HOME/lockgen/agent-ide/`** — never against `/mnt/c`, never `--package-lock-only`, never Docker (inherited from Gamify Decision 1). The exact invocation against Node 20.20.2 + Agent IDE's `package.json` is pinned empirically by Phase 1's walking skeleton — if explicit `--os`/`--cpu` passes are needed in addition, that surfaces there.
3. **Phase 4's regeneration preserves currently-resolved dependency versions** — no opportunistic minor bumps. Same anti-pattern Gamify Decision 3 codified (Contractor App's `stripe`/`recharts` churn). Mechanism: pin top-level caret ranges to currently-resolved exact versions before regen via `scripts/pin-toplevel.mjs` (adapted from Gamify's 4-manifest version to Agent IDE's single-manifest layout).
4. **The pre-push guard blocks any `package-lock.json` change not produced by `lockfile:sync`**; provenance via a `.lockfile-sync.marker` file containing `{lockfileSha256, generatedAt, generatedBy}` written by the wrapper, checked by the hook. Ships advisory-tunable per Gamify Decision 5.
5. **Stryker's `mutate` globs start tightly scoped to `src/shared/**/*.ts`** (pure types/helpers, no Electron imports, no React, no native bindings). Expansion candidates (`src/main/codebaseGraph/**` excluding the 3 xxhash importers; `src/main/**/*.helpers.ts`; `src/main/**/*.utils.ts`) are filed in `roadmap/follow-ups/2026-05-16-stryker-mutate-scope-expansion.md` for a future coverage-investment wave. Rationale: §3 of `research-92.md` — Stryker's sandbox cannot rebuild native modules ([stryker-js#1621](https://github.com/stryker-mutator/stryker-js/issues/1621)); the adapter pattern in Phase 2 reduces the leak surface but does not eliminate it. Tight globs are the safe v1; the follow-up captures the expansion candidates so they're not lost.
6. **Stryker's `break:` floor is set EMPIRICALLY in Phase 5** at the integer just below the first measured score on `src/shared/**`. If the first score is 73.2%, set `break: 73`. Anti-backslide only; raising it is a deliberate decision in a future coverage wave (Gamify Decision 6).
7. **`reports/stryker-incremental.json` is gitignored** (the one-time full re-baseline on a fresh clone is acceptable for a solo-dev repo) and Stryker runs dual-frequency in CI — incremental mode on PRs and pushes to master, a full `--force` run on a weekly Monday cron (Gamify Decision 7).
8. **`@node-rs/xxhash` is retained — it IS used.** 3 live imports in `src/main/codebaseGraph/` (`graphDatabaseSession.ts:9`, `indexingPipelineSupport.ts:6`, `mcpToolHandlerDefs.ts:6`), all `import { xxh3 } from '@node-rs/xxhash'`. Introduced Wave 14 (commit `60addbe6`) for codebase-graph content-hashing. The initial Phase 2 audit reported zero imports (false negative — grep missed the named-import shape); corrected before plan-lock. Stays in `package.json`; counts as the **fourth** native module Stryker's `mutate` globs must avoid (alongside `better-sqlite3`, `node-pty`, `@parcel/watcher`). Phase 2's adapter audit checks for further leaks outside `codebaseGraph/`.

## Risks

| Risk | Mitigation |
| ---- | ---------- |
| The npm 10.3+ pruning regression bites Agent IDE's `package.json` shape differently than Gamify's, and from-scratch install still yields an incomplete lockfile. | Phase 1's walking skeleton verifies completeness empirically (the win32/linux/darwin probe) *before* Phase 3 wraps the invocation. | Stryker's sandbox imports a native module transitively from `src/shared/**` and the baseline run fails before producing a score. | Phase 2's audit + Phase 6's tight `mutate` glob (`src/shared/**` only) is the primary defense.
