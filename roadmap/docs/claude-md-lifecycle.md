# CLAUDE.md Lifecycle

How CLAUDE.md files in this repo are generated, grown organically, and kept under the 200-line cap.

---

## Generation

### When generation runs

CLAUDE.md files are generated by `src/main/claudeMdGenerator.ts` via three triggers:

| Trigger | Mechanism |
|---|---|
| Manual (`claudeMd:generate` IPC) | User or agent calls the IPC handler in `src/main/ipc-handlers/claudeMd.ts` |
| Post-session (if `triggerMode: 'post-session'`) | `hooksSessionHandlers.ts` fires after `agent_stop` |
| Post-commit (if `triggerMode: 'post-commit'`) | `hooksSessionHandlers.ts` fires after a session that auto-committed |

The 3-minute cooldown (`COOLDOWN_MS = 180_000`) prevents multiple triggers from stacking up within the same session.

### Settings that shape output

Both live in the `claudeMdSettings` namespace of `electron-store` (defined in `src/main/configSchemaTail.ts`):

| Setting | Default | Effect |
|---|---|---|
| `leanMode` | `true` | Routes generation through the lean prompt template. When `false`, uses the legacy prompt. |
| `maxLines` | `200` | Target passed to the prompt as the size ceiling; also used by `scripts/claude-md-size-check.ts` to detect violations. |
| `model` | `'sonnet'` | Which Claude model is passed to the CLI via `--model`. |
| `enabled` | `true` | Master on/off. When `false`, all generation is skipped. |
| `excludeDirs` | `[]` | Relative paths excluded from generation. |
| `generateRoot` | `true` | Whether the project root gets a CLAUDE.md generated. |
| `generateSubdirs` | `true` | Whether subdirectories under `src/` are discovered and generated. |

## Lean prompt principles

The lean prompt is built by `src/main/claudeMdGeneratorLeanPrompt.ts` and enforces a hard separation between derivable content (which the codebase graph already serves) and tribal knowledge (which is CLAUDE.md's only job).

### EXCLUDE list

The following content types are explicitly excluded from the lean prompt:

- **File-role tables** ā€” `| File | Role |` tables listing every file in a directory. The graph's `get_architecture` / `search_graph` serves this on demand, always up to date.
- **Subdirectory indexes** ā€” lists of child directories with one-line descriptions. Same reason.
- **Import/export dependency lists** ā€” what the module imports or re-exports. `trace_call_path` in the graph.
- **Architecture flow diagrams** ā€” ASCII trees of component hierarchies. These go stale after every refactor.
- **Generic advice** ā€” "use error handling", "keep functions small" ā€” applies to any codebase and is therefore noise.

### INCLUDE list

Only tribal knowledge that cannot be derived from reading the code:

- Gotchas: load-bearing patterns that look wrong but must stay as-is.
- Design decisions with rationale ("we chose X because Y; do not refactor to Z").
- "Do not refactor this" warnings with the reason.
- Non-obvious invariants (ordering constraints, lifecycle sequencing, timing assumptions).
- Failure modes that are silent or hard to reproduce.
- Cross-cutting concerns that are invisible when reading files in isolation.

### "OMIT rather than speculate" rule

The lean prompt includes this directive verbatim:

> **OMIT rather than speculate.** If you are not certain a gotcha exists, leave the section empty. An empty "## Gotchas" section is correct. An invented gotcha is harmful.

This is the primary guardrail. Without it, models fill empty space with plausible-sounding but invented constraints that mislead future agents.

### Inline warnings as supporting evidence

`src/main/claudeMdGeneratorInlineWarnings.ts` scans `.ts` and `.tsx` files in the target directory for:

| Comment kind | Example |
|---|---|
| `NOTE` | `// NOTE: this must run before hydration` |
| `WARNING` | `// WARNING: calling this with null crashes the process` |
| `DO NOT` | `// DO NOT remove this ā€” required for Electron IPC teardown` |
| `HACK` | `// HACK: xterm internal, test after version bumps` |
| `ESLINT_REASON` | `// eslint-disable-next-line ... -- reason: path from validated root` |

These are quoted verbatim in the prompt as supporting evidence. The model is instructed to anchor gotchas in these comments rather than invent content. If no inline warnings exist, the prompt says so explicitly and instructs the model to leave `## Gotchas` empty.

### Why derivable content belongs in the graph, not CLAUDE.md

File-role tables become stale the moment a file is renamed, split, or deleted. A 30-file table in a CLAUDE.md that was generated 6 months ago is actively misleading ā€” some files no longer exist, new ones are absent. The codebase graph (indexed via `codebase-memory`) reflects the current state on every query. CLAUDE.md should only contain what the graph cannot know: the author's intent, the reason a pattern exists, and the failure modes discovered in production.

## Organic growth

### The gotcha-maintenance rule

When an agent discovers a non-obvious constraint, surprise behavior, or load-bearing pattern during work, it **must** append a line to the nearest subsystem CLAUDE.md's `## Gotchas` section before completing the task.

Format: `- **<topic>**: <rule>. Reason: <why>.`

This rule is defined in the project root `CLAUDE.md` and applies to all agents working in this repo.

### When the Stop-hook nudge fires

`src/main/hooks/gotchaUpdateNudge.ts` evaluates sessions at `agent_stop`. If the session looks like a bug-fix (existing files modified, commit message contains `fix`, `bug`, `issue`, `gotcha`, or `regression`), the hook emits a structured nudge suggesting the agent consider documenting the gotcha before closing the session.

The nudge is **passive** ā€” it does not block session completion or stash work. It records a telemetry entry regardless of agent follow-through so nudge-vs-follow-through rate can be measured over time.

### What an agent should do when nudged

1. Review the files modified in the session.
2. If any modification fixed a non-obvious constraint ā€” a timing bug, an undocumented invariant, a third-party workaround ā€” add a `## Gotchas` entry to the relevant subsystem CLAUDE.md.
3. Commit the addition in the same commit as the fix.

If nothing qualifies, dismiss the nudge without action.

## Size cap and lint

### The 200-line cap

Anthropic's guidance: **target under 200 lines per CLAUDE.md file**. Longer files reduce adherence ā€” models skim or skip sections. The lean prompt targets 150 lines to leave headroom for organic growth.

### `npm run lint:claude-md`

The script at `scripts/claude-md-size-check.ts` walks the repo and flags any CLAUDE.md over `claudeMdSettings.maxLines` (default 200) that does not contain the `<!-- claude-md-grandfathered -->` marker.

- Exit 0: all files within cap, or all over-cap files are grandfathered.
- Exit 1: at least one violation without the grandfather marker.

This script runs as a pre-commit hook (via `.husky/pre-commit`), so over-cap files block commits.

### The grandfather marker

A file that is temporarily over cap can be marked `<!-- claude-md-grandfathered -->` to pass the lint check while awaiting trim or regeneration. This is an escape hatch, not a long-term solution.

**At wave close, no grandfather markers should remain in the repo.** The wave process requires that all near-cap files be trimmed or regenerated during the wave rather than grandfathered.

### Pre-commit gating

The pre-commit hook runs `npm run lint:claude-md` before any commit. A violation blocks the commit and prints the violating file paths with their line counts.

## When to regenerate

Prefer regeneration when there has been a **material subsystem change** ā€” the directory's files have been substantially reorganized, new gotchas have accumulated since the last generation, or the existing CLAUDE.md is structurally stale (e.g., still uses old file-role table format throughout).

Git-diff-based stale detection is built into `claudeMdGeneratorSupport.ts` (`getChangedDirectories`). When `triggerMode: 'diff'`, generation runs only on directories containing files touched since the last generation.

Do NOT regenerate just to reduce line count. Use manual trim instead (see below).

---

## When to manually trim

Use manual trim for **small overshoots** (a few lines over cap) and **deduplication of legacy sections**.

Trim discipline:
- **Preserve**: gotchas, design decisions, inline-warning quotes, "do not refactor" warnings.
- **Drop**: file-role tables, subdirectory indexes, import/export dependency lists, architecture flow diagrams.

These dropped sections are graph-derivable. Removing them does not lose information ā€” `get_architecture` and `search_graph` return the same data on demand.

When a CLAUDE.md exceeds 200 lines, prefer manual trim over regeneration; only regenerate after a material subsystem change. This avoids the risk that regeneration invents new content to fill the space the trim would have removed.
