## Why this wave exists

### The problem

The user almost exclusively uses Opus 4.7 for chat. Opus dispatches Sonnet subagents on its own when the catalog routing rule (`~/.claude/rules/agent-catalog.md`) suggests doing so, but in practice Opus is **lazy about delegation** — it does mechanical work itself (reading 8 files, applying the same edit to 5 files, doing a clear haiku-explorer-shaped lookup) instead of dispatching the right specialized subagent or downshifting tier for sub-operations. The user explicitly said: _"It tends to be lazy about that, not that it is its fault, it is probably my fault but I am not educated enough on this content yet and it could be years before I precisely know when to use what or when to tell Opus to use what when."_

This wave builds the mechanism that compresses "knowing when to use what" into pattern-matched nudges, so the user doesn't have to learn the catalog and Opus can't quietly skip delegations.

### Why the existing model router doesn't solve this

The current `src/main/router/` subsystem is a 3-class tier classifier (HAIKU/SONNET/OPUS) for top-level chat prompts. It assumes the user is uncertain about model choice. The user is not uncertain — Opus is the chosen primary. The model router has nothing to optimize for this user, AND its training pipeline is structurally broken for this usage pattern (see the housekeeping section below).

The delegation coach is a **different subsystem** that operates _inside_ an Opus session, not at the chat boundary. It nudges Opus toward the catalog and toward cheaper tiers for sub-operations. The two subsystems are orthogonal; the coach does not replace the router (and the router can be left to run on bundled weights for any user who actually wants tier classification).

### Background context the next agent should know

The conversation that produced this spec went through these architectural alternatives before landing on the delegation coach:

1. **Heuristic prompt-shape labeler** — rejected as circular (the heuristic that judges "this prompt looks trivial → Haiku" _is_ the router; can't train a classifier on labels generated by another classifier of the same input).
2. **Outcome-shape heuristic labeler** — rejected because of the counterfactual problem. Observing that "Opus used 0 tools on this prompt" doesn't tell us Haiku could have done it; tier capability is asymmetric. Labeling counterfactuals from observations of a single tier produces biased labels that will lead the trained classifier to over-route to cheap tiers.
3. **LLM judge (`llmJudge.ts` referenced in `src/main/router/CLAUDE.md` but never implemented)** — would partially work but expensive and still doesn't address the "Opus is lazy mid-session" problem, which is the actual user pain point.
4. **Counterfactual sampling** — the only true ground-truth source for tier classification, but a wave's worth of work and still solves the wrong problem.
5. **Confidence-calibrated escalator (default Haiku, escalate)** — closer to the right shape, but doesn't match how the user works (they trust Opus and don't want their top-level chat downgraded).
6. **Delegation coach (this wave)** — operates inside an Opus session, observes tool-use patterns in real time, nudges toward catalog dispatch or cheaper tiers for sub-operations. Signal is observable directly (did Opus take the nudge; did the dispatched subagent succeed); no counterfactual problem.

The delegation coach is the design that matches how the user actually works.

## Goal

After this wave, when Opus is mid-session and about to do something that should have been delegated, a deterministic hook detects the pattern and:

- **Soft tier:** injects a reminder in Opus's next turn — "You've read 5 related files; consider haiku-explorer with a clear return shape for the rest."
- **Acknowledgment tier:** blocks the next tool call until Opus explicitly justifies why it's not delegating.
- **Hard-gate tier:** refuses the tool call and requires a catalog dispatch (with an explicit escape hatch for legitimate exceptions).

Each nudge + outcome is logged. Over time the data shows which patterns deserve which escalation level. The pattern library is curated, not learned — calibrated to the user's catalog and usage, not generic.

## Locked decisions

These came out of the conversation. The next agent can revisit them with the user, but they reflect the user's stated preferences and the architecture we agreed on.

1. **Coach lives outside the Electron main process.** The coach is implemented as Claude Code hooks at `~/.claude/hooks/`, not inside the IDE. Reason: hooks fire for all Claude Code sessions (terminal, IDE-internal, external) deterministically. Putting the coach in main would mean it only works inside the IDE chat, missing terminal sessions and external Claude Code use entirely.
2. **Pattern library is hand-curated and version-controlled.** Stored as a JSON or TypeScript file in the agent-ide repo (proposed: `src/main/delegationCoach/patterns.ts`) so the IDE can read it for analytics, AND symlinked or copied into `~/.claude/hooks/lib/coach-patterns.json` for the hooks to consume. **Not** machine-learned. The point is to encode the user's catalog explicitly, not to discover patterns from scratch.
3. **Escalation tiers ship in three phases, soft-first.** First cut is soft-nudge only across all patterns. Promote individual patterns to acknowledgment-required or hard-gate based on observed data, not upfront. Avoids day-one friction.
4. **Coach is self-disabled inside subagent sessions.** Subagents have already been delegated to; they should not be coached to delegate further (and the harness blocks them from spawning sub-subagents anyway). Detect via the `<SUBAGENT-STOP>` pattern or environment marker the harness sets when a subagent is running.
5. **Storage location: `{userData}/delegation-coach.jsonl`.** Same shape and lifecycle as `router-decisions.jsonl`. Lives under Electron's userData so the IDE can read it for analytics; hooks write to it directly via filesystem (no IPC needed). Path: `%APPDATA%/ouroboros/delegation-coach.jsonl` on Windows; per-OS equivalents elsewhere.
6. **Housekeeping: kill the model router auto-retrain loop in this wave.** It's actively wasteful (every 30s spawns Python, fails validation, repeats — see "Housekeeping" section below for the full diagnosis). The fix is small and tangentially related; bundling avoids leaving known-broken code running.

### Open questions for the next agent to confirm with user

- **Q1.** Should the coach also fire for chat-prompt-level model selection (i.e., suggest "this prompt looks trivial; consider switching to Sonnet")? Default proposed: **no.** The user has decided "Opus by default" and doesn't want the coach second-guessing that. Coach is for sub-operations only. Confirm.
- **Q2.** What's the seed pattern library? The conversation produced a draft (see Phase A below) but the right way to ground it is to walk a real recent session and identify gaps. Confirm whether to do the audit or ship the draft and tune later.
- **Q3.** Hook performance budget. The PreToolUse hook runs on every tool call; pattern matching needs to stay <10ms. If the pattern library grows past ~50 entries the matching logic may need indexing. Acceptable to ship without indexing if the initial library is small. Confirm.

## Scope

### In-scope

- **Pattern library module.** `src/main/delegationCoach/patterns.ts` — hand-curated table mapping detected pattern → suggested catalog entry / cheaper tier. Initial seed of 8-10 patterns (listed in Phase A).
- **Pattern detector.** Pure function: `(toolUseStream, currentToolCall) → MatchedPattern[]`. Stateless if possible; if state is required (e.g., counting recent Reads), state is held in the hook process and bounded.
- **PreToolUse hook integration.** New hook file `~/.claude/hooks/delegation_coach.mjs` chained after the existing `pre_tool_use.mjs` and `agent_catalog_enforce.mjs`. Soft-nudge mode only in first cut.
- **Storage layer.** Append-only JSONL at `{userData}/delegation-coach.jsonl` with rotation policy matching `router-decisions.jsonl`. Schema: `{ timestamp, sessionId, pattern, escalation, toolCall, opusResponse, outcome, meta }`.
- **PostToolUse / SubagentStop integration** — outcome tracking. When a nudge fires and Opus subsequently dispatches a subagent, log the dispatch as the outcome. When Opus ignores the nudge and proceeds, log that too.
- **Acknowledgment tier mechanism.** PreToolUse hook returns exit 2 with reason on stderr; Claude Code treats this as a soft block requiring Opus to respond. The hook can be promoted from soft to acknowledgment per-pattern via a config flag in `patterns.ts`.
- **Hard-gate tier mechanism.** Same as acknowledgment but the hook output also includes the explicit dispatch suggestion as a directive. Includes escape-hatch protocol: Opus can include `[delegation-bypass: <reason>]` in the next message to override.
- **Housekeeping fix to retrain loop.** See dedicated section. Two changes in `src/main/router/retrainTrigger.ts`: detect "trainer succeeded but skipped" and advance the counter; OR add a `routerAutoRetrainEnabled` flag (default false). Pick one.
- **Tests.** Unit tests for pattern matcher (synthetic tool-use streams), integration tests for the hook (mock Claude Code stdin/stdout), test for the storage writer.
- **Docs.** New `src/main/delegationCoach/CLAUDE.md`. Update `src/main/CLAUDE.md` subsystem map. Update `src/main/router/CLAUDE.md` to note that the model router is in maintenance-only mode (bundled weights, no auto-retrain) and the coach is the active model-selection feature. Update `~/.claude/rules/agent-catalog.md` to note the coach now enforces what the rule used to merely suggest.

### Out-of-scope

- **Machine learning over the coach data.** Out of wave. The data is collected for analytics and manual tuning. No classifier trained from it. If we ever do, it's a follow-up wave.
- **LLM judge (`llmJudge.ts`).** Out. The coach is deterministic by design; no LLM call in the hot path.
- **Counterfactual sampling.** Out. Not relevant for the coach's signal model.
- **Migration of model-router data.** The model-router subsystem stays in place, just dormant. We don't delete or migrate `router-decisions.jsonl` or `router-quality-signals.jsonl`.
- **Coach for non-tool decisions.** The coach observes tool calls only. It does not nudge Opus on response style, message length, or other non-tool behaviors.
- **GUI for managing the pattern library.** Patterns are edited as code. No settings panel.
- **Renaming the model router subsystem.** Stays as `src/main/router/`. The coach is `src/main/delegationCoach/`.

## Housekeeping (Phase 0 prerequisite — must ship)

The model-router auto-retrain loop is currently broken in a way that wastes resources every 30 seconds indefinitely. This wave fixes it because we're touching the same conceptual area; not fixing it leaves known-broken code running.

### The bug

`src/main/router/retrainTrigger.ts:72-89` (`checkAndRetrain`) fires every 30 seconds. When `currentSignalCount - lastRetrainCount >= 50`, it triggers a retrain via `runRetrainPipeline`. The pipeline:

1. Exports decisions+signals (`routerExporter.ts`) → writes `router-full-extracted.jsonl` and `router-full-judged.jsonl` (logs `[exporter] wrote N extracted, M judged`).
2. Spawns `tools/train-router.py` via `spawnTrainer`.
3. Trainer hits the `MIN_PER_CLASS_SAMPLES = 5` floor (`tools/train-router.py:455-474`) — needs ≥5 samples per HAIKU/SONNET/OPUS class. Because the user always picks Opus, signals derive an all-OPUS label distribution (see "Why labels are all-OPUS" below). Trainer **exits cleanly with `sys.exit(0)`** without writing the output file.
4. Back in TS, `spawnTrainer` returns `success: true` (exit 0).
5. `validateWeightFile(outputPath)` tries to read `router-weights-retrained.json` — file doesn't exist → returns false.
6. Logs `[retrain] output weights invalid — keeping old weights`.
7. **`setLastRetrainCount(signalCount)` is never called** because it's gated on `loaded === true` (line 124).
8. 30 seconds later, the same 203 (or whatever) signals trigger the same wasted cycle. Forever.

Observed in user's log over a 5-minute window:

```
23:32:05.739 > [retrain] triggering retrain (203 signals)
23:32:05.746 > [exporter] wrote 106 extracted, 27 judged
23:32:07.741 > [retrain] output weights invalid — keeping old weights
[repeats every 30s]
```

### Why labels are all-OPUS

`src/main/router/routerExporterHelpers.ts:45-61` (`signalToLabel`) maps signals to labels in only two ways:

- **Positive signals** (`code_committed`, `task_completed`, `terminal_natural_stop`) → label = `tier_used` (reinforce).
- **Negative signals** (`user_abort`, `chat_regenerate`, `chat_correction`, `task_interrupted`) → label = `tierUp(tier_used)` (escalate).
- **`user_override`** → label = whatever the user manually picked.

There is **no path that produces a de-escalation label.** And `tierUp('OPUS')` returns `'OPUS'` (line 17), so even aborted Opus sessions are labeled OPUS. With the user's "always Opus" usage pattern, ~100% of labels are OPUS, ~0% are HAIKU or SONNET. The trainer's per-class floor will never be reached organically.

### The fix

Pick one of these two options and ship it. (Coach work proceeds regardless; this is independent and small.)

**Option H1 — Disable auto-retrain by default.** Add `routerAutoRetrainEnabled: boolean` to the router config in `configSchema*.ts`, default `false`. `observeDatasetGrowth` short-circuits when the flag is false. `loadRetrainedWeightsIfAvailable` continues to load any pre-existing retrained weights file. Rationale: simplest; preserves data collection for any future analytics; user can flip the flag if/when retraining becomes useful.

**Option H2 — Advance counter on clean-skip.** When `spawnTrainer` returns `success: true` but `validateWeightFile` returns false, treat it as "trainer skipped due to insufficient data" and call `setLastRetrainCount(signalCount)` anyway. Optionally detect the trainer's "Skipping training: insufficient data" stdout marker to confirm it really was a skip and not a silent crash. Rationale: keeps the loop functional in case data ever becomes balanced; no config change.

**Recommendation: H1.** The coach makes the model router structurally redundant for this user; turning auto-retrain off by default reflects that. H2 is a band-aid on a system that's not the active model-selection mechanism anymore.

The user can be asked which to take during Phase 0; if they don't have an opinion, ship H1.

### What this housekeeping does NOT do

- Does not delete `src/main/router/`. The classifier still serves bundled weights for any user who wants tier routing.
- Does not stop signal/decision logging. `router-decisions.jsonl` and `router-quality-signals.jsonl` continue to be written.
- Does not implement `llmJudge.ts`. The reference in `src/main/router/CLAUDE.md` should be updated to "deferred — see Wave 61 ADR" or removed.

### Phase A — detector module shape (sketch)

```typescript
// src/main/delegationCoach/types.ts
export interface ToolCallEvent {
  tool: string;
  input: Record<string, unknown>;
  timestamp: number;
  sessionId: string;
}

export interface PatternMatch {
  patternId: string;
  suggestion: string;
  escalation: 'soft' | 'acknowledgment' | 'hard';
  confidence: number; // 0-1, used to suppress noisy matches
}

// src/main/delegationCoach/detector.ts
export function detectPatterns(
  history: ToolCallEvent[], // recent tool calls in this session, bounded window
  current: ToolCallEvent, // the call about to fire
  patterns: PatternDefinition[],
): PatternMatch[];
```

History window: last 60 seconds OR last 20 tool calls, whichever is shorter. Bounded so memory cost is constant.

## Out-of-wave follow-ups

- **Coach pattern auto-discovery.** Mining the JSONL for high-frequency tool-use sequences that _aren't_ in the pattern library, surfacing them as candidates. Wave-sized; do after Phase F has produced enough data to make the discovery meaningful.
- **`llmJudge.ts` for the model router.** If the user ever decides they want tier classification despite the coach, this is the missing piece. The reference in `src/main/router/CLAUDE.md` documents it but the file doesn't exist. Build only when there's a concrete need.
