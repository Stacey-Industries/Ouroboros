## 2. Integration shape options

**Option A - call `preparePendingSend` + `OrchestrationAPI` from the new handler.**  
Surface: mostly `chatStateNewPath.ts`, preload/shared payload types, tests. The handler would validate the enriched payload, resolve options, call `preparePendingSend`, register the turn using the eventual task id, then call `createTask`/`startTask`. Risk: medium-high unless the shadow tap registration is explicit. `wrapWithShadowTap` forwards stream-json to `getShadowTap()?.onStreamJsonEvent` (`src/main/orchestration/providers/claudeCodeLaunch.ts:8-17`), but the tap has its own private registry (`src/main/agentChat/dualEmitOrchestrator.ts:41-55`) and only `onCommand` registers a turn (`src/main/agentChat/dualEmitOrchestrator.ts:98-116`). Therefore wrap-only is insufficient; the new path must either call `tap.onCommand` before `startTask` or retire the tap in favor of direct shared-registry dispatch. Alignment: acceptable as a bridge activation step, but it keeps the legacy orchestration facade semantics alive.

**Option B - duplicate TaskRequest/settings logic in the new path.**  
Surface: large edits in `chatStateNewPath.ts` plus duplicated helpers for settings, thread resolution, context/history, attachments, and provider dispatch. Risk: high. It avoids coupling to files slated for deletion, but duplicates logic already covered by tests and likely diverges on provider-specific rules such as Codex permission clamping (`src/main/agentChat/chatOrchestrationRequestSupport.ts:218-230`) and resume invalidation (`src/main/agentChat/chatOrchestrationRequestSupportHelpers.ts:216-238`). Alignment: weak; it creates a second preparation source while Wave 86 requires one canonical owner and hard-fail boundaries (`docs/superpowers/specs/2026-05-11-chat-orchestration-state-architecture-design.md:37-44`).

**Option C - lift send preparation/provider dispatch into shared modules consumed by both paths.**  
Surface: extract `preparePendingSend`, send-option resolution, provider dispatch, attachment/context packet path, and git pre-snapshot into a non-bridge `chatSendCoordinator`/`chatTaskDispatch` module. Legacy bridge calls the module until Phase 3; new handler calls it now and routes canonical events through shared `registry`/`broadcaster`. Risk: medium. It has refactor risk, but minimizes behavior drift and makes Phase 3 delete bridge glue rather than the send pipeline. Alignment: strong: one preparation pipeline, one identity registry, one state-machine mutation surface.

**Option D - new-path coordinator, legacy bridge becomes a temporary adapter.**  
Surface: add `src/main/agentChat/chatSendCoordinator.ts` that accepts an `AgentChatSendMessageRequest`, owns validation/preparation/task dispatch/cancel bookkeeping, emits canonical command/stream/terminal/commit events directly to `broadcaster`, and exposes a thin compatibility method for the legacy bridge during Phase 2. Files edited: `chatStateNewPath.ts`, preload/shared chat-command types, `chatOrchestrationBridge.ts` adapter call site, tests. Risk: medium-high up front, lowest long-term. Alignment: strongest: it makes the coordinator state-machine-first instead of bridge-first, while reusing preparation/provider helpers rather than cloning them.

## 3. Recommended option + rationale

Pick **Option D**, implemented by extracting shared preparation/provider dispatch as in Option C but making the new coordinator the owner. The architecture spec says main owns thread records, message list, send queue, thread status, identity registry, and active turn metadata (`docs/superpowers/specs/2026-05-11-chat-orchestration-state-architecture-design.md:64-72`); all event sources must flow through `EventNormalizer -> ChatSessionStateMachine -> ChatStateBroadcaster` (`docs/superpowers/specs/2026-05-11-chat-orchestration-state-architecture-design.md:137-164`); and the state machine is the only mutation surface (`docs/superpowers/specs/2026-05-11-chat-orchestration-state-architecture-design.md:338-342`). Option D matches that target while preserving proven legacy behavior through reused helpers. It also respects Decision 2's renderer-driven seam (`roadmap/wave-87-chat-orchestration-activation/wave-87-decisions.md:26-34`) and avoids new IPC channels; enriching `chatCommand:sendMessage`'s payload is type expansion on an existing channel, not a new channel (`roadmap/wave-87-chat-orchestration-activation/wave-87-decisions.md:70-78`).

## 4. Phase 2 (rebuilt) plan

**Phase 2A - main-process build-out.** Enrich `ChatCommandPayload` beyond `{ threadId, content }` (`src/main/agentChat/eventNormalizer.ts:47-50`) to match `AgentChatSendMessageRequest`: optional `threadId`, `workspaceRoot`, `content`, attachments, context selection, overrides, metadata, and skill expansion (`src/shared/types/agentChat.ts:363-372`). Add the coordinator. It should call validation/settings/preparation, capture pre-snapshot, register the canonical turn/task identity, dispatch `turn_submitted`, call provider dispatch through `createMinimalOrchestration`, use direct stream events or shadow tap only after registry registration, dispatch terminal plus `message_committed`, retire aliases, and preserve cancel bookkeeping. Estimated 5-8 files, +300 to +600 LOC, high judgment.

**Phase 2B - renderer cutover.** Reapply the prior renderer migration against the now-complete handler: `sendAgentChatRequest` calls `window.electronAPI.chatCommand.sendMessage` with the existing `SendRequest` shape, hydrates via `chatState:requestSnapshot`, and preserves queue/draft/optimistic behavior. Estimated 3-5 files, +80/-120 LOC, medium judgment.

Make these **sequential**. 2B cannot prove behavioral parity until 2A owns the real send pipeline. A single dispatch would mix main boundary construction with UI rebinding and obscure failures in the orchestrator-owned Phase 2 acceptance test (`roadmap/wave-87-chat-orchestration-activation/wave-87-decisions.md:60-68`).

## 5. Risks + open questions

- State extension: canonical events currently carry `turn_submitted.content` only (`src/shared/types/canonicalChatEvent.ts:40-49`). If model/effort/permissionMode/preSnapshotHash are canonical state, add metadata to canonical events or persist it through coordinator-owned records. Do not reintroduce sticky merge rules; those are explicitly a leak class (`docs/superpowers/specs/2026-05-11-chat-orchestration-state-architecture-design.md:53-58`).
- Persistence: `ChatPersistenceLayer` notes `message_committed` as the turn commit boundary (`src/main/agentChat/chatPersistenceLayer.ts:161`), but the current new handler never dispatches that event after `turn_completed`; it only retires aliases on process result (`src/main/ipc-handlers/chatStateNewPath.ts:158-169`).
- Attachments: payload must carry `ImageAttachment[]`; providers already materialize images for Claude Code and Codex (`src/main/orchestration/providers/claudeCodeLaunchInputs.ts:28-40`; `src/main/orchestration/providers/codexAdapterLaunchSupport.ts:141-146`).
- Cancel: legacy has `cancelTask` and `cancelByThreadId` IPC (`src/shared/types/agentChatResults.ts:203-205`) and provider adapters can cancel (`src/main/ipc-handlers/agentChatOrchestration.ts:267-275`). New channel constants have no cancel command (`src/shared/ipc/chatStateChannels.ts:13-40`) even though canonical events model `turn_cancelled` (`src/shared/types/canonicalChatEvent.ts:212-221`). Decision 6 forbids adding a channel without Cole approval.
- Do not carry forward `inferSessionId` or `applyStickyLinkFields`; Phase 0 schedules both for deletion (`roadmap/wave-87-chat-orchestration-activation/phase-0-results.md:13-35`) and the spec names them as architecture leaks (`docs/superpowers/specs/2026-05-11-chat-orchestration-state-architecture-design.md:53-58`).
